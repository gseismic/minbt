# PLAN-023 市场与退出生命周期

## 背景

复核确认以下问题真实存在：

1. T+1 且允许做空的市场中，锁定多头可以通过跨零反手订单绕过可平数量检查。
2. 已平仓订单在同一标的重新开仓后仍可调用 `set_exit()`，旧订单会错误控制新持仓。
3. 新退出配置覆盖当前持仓时，旧 `ExitConfig.active` 仍可能保持 `True`。
4. callable `state` 每次检查都会重新调用，无法保存自定义退出条件状态。
5. 公开 `ExitConfig` 缺少 `symbol/portfolio`，并暴露内部可变 `ExitRule` 列表。

## 目标

1. 所有反向订单都必须先满足实际平仓部分的可用数量约束。
2. `order_id` 只能控制其所属的当前净持仓生命周期。
3. 公开退出配置准确反映实际激活状态，不泄漏内部可变规则对象。
4. 自定义退出状态在多个回测时间点之间持久化。

## 实施方案

1. Market 对反向订单计算实际平仓数量 `min(abs(qty), abs(current_size))`。
2. 实际平仓数量超过 `available_size` 时拒绝订单，包括跨零反手。
3. Broker 按 `(portfolio, symbol)` 维护当前持仓生命周期内有效的订单 ID 集合。
4. 从空仓开仓时创建新生命周期；同方向调整时加入当前订单；全平或反手时结束旧生命周期。
5. `set_exit()` 和 `add_exit()` 校验订单属于当前持仓生命周期。
6. 新退出配置成为当前配置时，将旧配置同步标记为 inactive。
7. callable state 在 `add_exit()` 时调用一次并保存结果字典。
8. 引入内部退出状态结构保存 `ExitRule` 对象。
9. 公开 `ExitConfig` 补齐 `symbol/portfolio`，`custom_rules` 只返回规则名元组。
10. `get_exit()`、`set_exit()`、`clear_exit()` 返回公开配置快照，避免外部修改内部状态。

## 测试

1. 当日锁定多头无法通过超量卖出反手为空头。
2. 解锁后允许合法平仓或反手。
3. 空头反手为多头时只锁定新增多头净额。
4. 平仓后重新开仓，旧订单调用 `set_exit/add_exit` 均失败。
5. 同一持仓生命周期的新退出配置覆盖旧配置，旧配置查询为 inactive。
6. callable state 工厂只调用一次，状态连续两次检查后触发退出。
7. 多头、空头、部分减仓、全平和反手生命周期均有覆盖。
8. `ExitConfig` 字段和类型与设计一致。

## 验收

1. T+1 不存在跨零绕过路径。
2. 退出订单、当前持仓和 `get_active_order()` 保持一致。
3. 公开退出配置不可直接修改 Broker 内部规则状态。
4. 聚焦测试与全量测试通过。
