# PLAN-022 Broker 事务原子性

## 背景

对照 `docs/design/minbt-20260630-system-design.md` 复核当前实现后，确认以下问题真实存在：

1. `close_portfolio()` 只预检市场规则，逐笔执行时后续订单仍可能被 Portfolio 拒绝，导致部分平仓。
2. 限价单提交时不做资金预检，明显无法成交的订单仍进入 pending。
3. pending 限价单的退出条件在成交后通过 `set_exit()` 校验；持仓方向在挂单期间变化时，会出现订单已成交、随后抛异常、退出配置未激活的半完成状态。

## 目标

1. `close_portfolio()` 满足全成或全不成。
2. 限价单提交时完成当前账户状态下的资金预检。
3. 限价单触发、成交和退出条件激活保持状态一致。
4. 任何预期内拒绝都不得留下部分成交、残留 pending 元数据或孤立退出配置。

## 实施方案

1. 为 Portfolio 增加非变更式订单预检能力，通过状态副本顺序模拟订单。
2. `close_portfolio()` 先完成全部 Market 校验和 Portfolio 顺序预检，再执行真实订单。
3. 限价单提交时使用 `limit_price` 和订单 leverage 预检 Portfolio。
4. MVP 只做提交时资金快照校验，不锁定或预留 pending 订单资金。
5. 限价单提交阶段只校验退出参数的数值、范围和互斥关系。
6. 限价触发时，根据预计成交后持仓和 `limit_price` 做方向校验。
7. 触发阶段退出条件不再有效时，将限价单原子地更新为 `rejected`，不执行成交。
8. 成交成功后直接激活已验证的退出配置，不调用面向中途修改语义的 `set_exit()`。
9. 清理所有 terminal pending 订单对应的 leverage 和退出参数元数据。

## 测试

1. 高手续费、多仓位组合关闭时，第二笔资金不足不会导致第一笔先成交。
2. 原子关闭成功后，全部持仓关闭，子组合现金转回 `main`。
3. 限价单提交时资金不足直接返回 `rejected`。
4. 限价单提交后资金被其他订单占用时，触发阶段返回 `rejected` 且不成交。
5. 限价单挂起期间持仓方向改变，退出参数失效时不抛出成交后异常。
6. 有效退出参数在限价单成交后正常激活。
7. 被拒绝或取消的 pending 订单不残留内部元数据。

## 验收

1. 聚焦测试通过。
2. 全量测试通过。
3. 失败场景中的现金、持仓、订单、pending 元数据和退出配置互相一致。
4. 代码 review 未发现新的部分提交路径。
